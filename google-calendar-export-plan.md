# Google Calendar Export — Plan

## Top-Level Overview

Build a **pure client-side Chrome extension (Manifest V3)** using **WXT + React + TypeScript** that lets a user:
1. Sign in with their Google account via `chrome.identity.launchWebAuthFlow` (OAuth 2.0)
2. Pick a date in the extension popup
3. Preview their Google Calendar events for that date
4. Export those events (title, start time, end time, location, description, attendees, status) directly to a Google Sheet — either a brand-new spreadsheet or a new tab inside an existing one named after the chosen date

No backend server. No local CSV download. All Google API calls are made directly from the extension using the user's OAuth access token stored in `chrome.storage.local`.

---

## Architecture Overview

- **Framework**: WXT + React + TypeScript (Chrome Manifest V3)
- **Auth**: `chrome.identity.launchWebAuthFlow` — OAuth 2.0 PKCE flow, access token stored in `chrome.storage.local`
- **Token refresh**: handled in the background service worker via `chrome.identity.getAuthToken` or manual refresh token exchange
- **Google OAuth scopes**:
  - `https://www.googleapis.com/auth/calendar.readonly`
  - `https://www.googleapis.com/auth/spreadsheets`
  - `https://www.googleapis.com/auth/drive.readonly`
- **Google API calls**: made directly from the popup (or via message passing to the background worker) using `fetch()` against the Google REST APIs with `Authorization: Bearer {token}`
- **Extension entry points** (WXT conventions):
  - `entrypoints/popup/` — React popup UI
  - `entrypoints/background.ts` — service worker for auth token management
- **Storage**: `chrome.storage.local` for the OAuth access token and refresh token

---

## Data Flow

```
User clicks extension icon
  → Popup opens
  → Checks chrome.storage.local for token
  → If no token: triggers chrome.identity OAuth flow via background worker
  → Token stored in chrome.storage.local
  → User picks a date
  → Popup calls Google Calendar REST API directly with Bearer token
  → Events displayed in table
  → User picks destination (new sheet or existing)
  → Popup calls Google Sheets + Drive REST APIs directly
  → New tab created in spreadsheet named after the date
  → Success banner with link to open the sheet
```

### Sub-Task 3 — Auth Service (Background Worker)

**Intent**: Implement token acquisition and storage in the background service worker so the popup can request a valid access token without managing OAuth itself.

**Expected Outcomes**:
- Background worker exposes a message handler that returns a valid access token when the popup sends a `GET_AUTH_TOKEN` message
- If no token exists or it is expired, triggers `chrome.identity.getAuthToken` to get a fresh one
- Token is cached in `chrome.storage.local`
- A `SIGN_OUT` message handler revokes the token and clears storage

**Todo List**:
1. Create `entrypoints/background.ts`
2. Add a `chrome.runtime.onMessage` listener handling two message types:
   - `GET_AUTH_TOKEN`: call `chrome.identity.getAuthToken({ interactive: true })`, store the token in `chrome.storage.local`, and return it to the popup
   - `SIGN_OUT`: call `chrome.identity.removeCachedAuthToken` + `chrome.identity.revokeAuthToken`, clear `chrome.storage.local`
3. Create `utils/auth.ts` in the popup — helper functions `getAuthToken()` and `signOut()` that send messages to the background worker and return the result

**Relevant Context**:
- `chrome.identity.getAuthToken` with `{ interactive: true }` shows the Google sign-in UI if no token is cached
- Background service workers in MV3 are ephemeral — do not store token only in memory; always persist to `chrome.storage.local`
- Message passing between popup and background uses `chrome.runtime.sendMessage`

### Sub-Task 4 — Google API Client Utilities

**Intent**: Create thin, typed wrapper functions around the Google Calendar, Drive, and Sheets REST APIs so the popup components have a clean interface and all `fetch` + auth header logic is centralised.

**Expected Outcomes**:
- `utils/calendar.ts` exports `getEventsForDate(token, date)` → typed event array
- `utils/drive.ts` exports `listSpreadsheets(token)` → `{ id, name }[]`
- `utils/sheets.ts` exports `exportToNewSheet(token, date, events)` and `exportToExistingSheet(token, spreadsheetId, date, events)` → `{ spreadsheetUrl }`
- All functions throw a typed error on API failure

**Todo List**:
1. Create `utils/calendar.ts`:
   - Build `timeMin`/`timeMax` from the date string
   - `GET https://www.googleapis.com/calendar/v3/calendars/primary/events` with `singleEvents=true`, `orderBy=startTime`
   - Map response to `CalendarEvent[]` type: `{ id, title, startTime, endTime, location, description, attendees, status }`
   - Handle all-day events (`start.date` vs `start.dateTime`)
2. Create `utils/drive.ts`:
   - `GET https://www.googleapis.com/drive/v3/files` with mime type filter and `orderBy=modifiedTime desc`
   - Return `{ id: string, name: string }[]`
3. Create `utils/sheets.ts`:
   - `exportToNewSheet`: `POST https://sheets.googleapis.com/v4/spreadsheets` to create, then `PUT .../values/{range}` to write
   - `exportToExistingSheet`: `POST .../batchUpdate` to add a tab named after the date (with collision suffix if needed), then `PUT .../values/{range}` to write
   - Shared `buildRows(events)` helper that produces the 2D array (header + data rows), joining attendees with `"; "`
4. Define shared types in `types/index.ts`: `CalendarEvent`, `Spreadsheet`

**Relevant Context**:
- All requests need `Authorization: Bearer {token}` header
- `valueInputOption: "USER_ENTERED"` for Sheets write so times/dates are parsed correctly
- Tab name collision: if a tab named `2025-07-14` already exists, use `2025-07-14 (2)`, etc. — check existing sheet titles from the `spreadsheets.get` response before calling `batchUpdate`
